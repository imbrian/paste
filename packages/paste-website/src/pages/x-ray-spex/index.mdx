## By default: just use Paste

For all internal Twilio products, Paste can be used to build accessible, consistent, and high quality customer experiences that require zero customization. You can still [roll your own custom components](https://paste.twilio.design/tokens/how-to-compose-custom-ui-with-tokens) with the Paste [primitive](https://paste.twilio.design/primitives) and [token](https://paste.twilio.design/tokens) system for product-specific components or one-off implementations. We recommend using this approach when:

- You are building default experiences that have not been customized.
- Customers do not need to customize the UI

Users are limited to style props and variants we ship for each component.

## Customization options unlocked by CustomizationProvider

To enable Customization for end customers while still maintaining an option for out of the box Paste, we need to provide a path that is clearly separated from the core system. By doing this, customization is additive and not directly at the component level. This enables easier component upgrades and a clear separation of concerns.

In this situation we might distribute a `CustomizationProvider`, strictly for use in interfaces where customers may need customizations. This could come with a Flex theme or with the upcoming Unified Design Language for Twilio.

To enable customization, wrap your application, plugin or custom component in the `CustomizationProvider`.

In the example below, the `CustomizationProvider` is using the Paste default theme, which is a preview of the upcoming Unified Design Language for Twilio. Things like font and colors have changed from the first example.

## Customization strategy 1 - Customization via a global theme

The `CustomizationProvider` allows you to override or add to the default theme. There is a defined [Theme shape](https://paste.twilio.design/tokens/theme-package) that tie groups of key value pairs to specific styling properties. `borderColors`, `backgroundColors`, `TextColors` etc. Each Key value pair, is a named Design Token and corresponding value [provided by Paste](https://paste.twilio.design/tokens).

You can probably add new tokens to the theme, and as long as they are in one of the pre-existing styling prop categories they _should_ work, though this isn't really tested here. We recommend using this approach when:

- As a Twilio customer, I would like to take my brand signals or style guide (colors, typeface, font-size, line-height) and apply it across the product globally
- I do not need the design opinions of Paste components to be changed (e.g. changing the border-radius of a text input, or rounding a specific button)

## Customization strategy 2 - Style specific components globally

This strategy goes a level deeper than a global theme. It can be applied in addition to stratetgy 1, allowing for a custom theme, and a customized component.

Overriding the theme can only get you so far. There are design details encapsulated in the component that are not exposed at the theme level, such as what radius, or background color token is being used within the component. You may not wish to change `borderRadius20` globally, just to have a 5px radius on cards. By using this approach you can break those design opinions only at the component level.

We recommend using this approach when:

- As a Twilio customer, I would like to globally customise all cards using either token values from a default or custom theme, or arbitrary values.

In the example below, we have changed all cards to have a different, border, shadow and hover treatment.

## Customization strategy 3 - Custom component variants

This customization strategy is similar to the option 2. Here, you can target component variants globally. Again, this strategy can be used in addition to the previous approaches, helping customers build a more comprehensive and robust UI system.

In the example below, we have targeted the error variant of the Alert component, and the inbound variant of the custom chat log component.

## Customization strategy 4 - Customize specific instances of a component

The final layer of customization targets single instances, entirely custom components, and includes variants and tokens. We recommend using this approach when:

- As a Twilio customer, I would like a specific button (End Call) to be styled with my brand purple while keeping all other buttons unchanged.
- I would like to style specific chat bubble with a yellow background for chat bot responses.
